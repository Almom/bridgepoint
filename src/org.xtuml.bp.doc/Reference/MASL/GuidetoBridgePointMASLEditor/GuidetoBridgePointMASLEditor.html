Guide to BridgePoint MASL Editor

1 Introduction

In BridgePoint MASL, the graphical capabilities developed to support the entry of xtUML are adapted to MASL to provide new views into the MASL model data.

The conversion and import steps transform textual MASL into xtUMLs mixture of graphical views of structure and textual views of actions. MASL activities align with xtUML action blocks while MASL components, domains and projects map very well with xtUML components and packages.

Consequently when viewing or modifying MASL domains, actions or projects, the BridgePoint Editors can be employed.

2 The xtUML Editor Perspective

When working with BridgePoint MASL workspaces, the xtUML Editor perspective should be used exclusively.

This perspective consists of five primary views: Model Explorer, Editor Canvas, Palette, Console, and Properties

Each of these views are documented in the BridgePoint MASL Help facility which can be accessed through Help>Help Contents.

A popular addition to the default xtUML Editor perspective is the Navigator view which can be added by Windows>Open Perspective>

Each canvas has an Editor that can be opened by selecting a model element in the Model Explorer view and either <double-click>ing on it or <RMB>>Open With which presents a list of editor options and hover over the desired editor selects that choice. Once selected, a new tab will appear in the canvas area and the will open containing the model element.

3 BridgePoint MASL Editor

MASL activities perform a similar role to OAL in xtUML models. However, the syntax is different and the OAL editor offers limited benefit when editing MASL. Consequently, in BridgePoint MASL there is an additional editor specifically for the viewing and modifying of MASL activity. The editor is a full featured text editor with MASL syntax highlighting and is denoted from the OAL editor by the black raven in the Tab of the editor window.  

The editor can be open by either double clicking an activity that contains only MASL

Select the activity in the Model Explorer than <RMB>> Open MASL

Or <dbl clicking> a .masl file in the Navigator view

3.1 When an Activity contains both OAL and MASL

In this case, from the Model explorer the Open With selection will present both OAL and MASL editor options. Hovering over a selection will open either the OAL or MASL depending on the editor chosen

The second alternative is to use double click with the preference selected as MASL. The preference is located under Window>Preferences>xtUML (check).

4 MASL Import using UI

To import a MASL domain or MASL project involves three steps: create the xtUML project, enable the use of Inter-Project references, and finally use import>xtUML model to bring the masl2xtuml created xtUML model file into the xtuml project.

It is important to note that the top-level xtUML project translated from a MASL project only after xtUML projects have been created and translated MASL domains imported for all domain projects referenced.

5 Inter-Project References

Inter-Project References (IPRs) are a mechanism for xtUML model elements defined in one xtUML project to be used in another xtUML project.

For the reference to be established both the referring project and the referenced project must have IPRs enabled

To check if IPRs are enabled for an xtUML project, select that project in the Model Explorer view, then select <RMB>> Preferences. This will open a popup window and selecting Interr-Project References will show two settings :   A and B. The use of IPRs is enabled if the checkbox beside A is marked. For all xtUML projects involved in a BridgePoint MASL model must have this checkbox marked.

When a domain component, shared datatype or shared interface is modified, all projects referencing that model element will be marked with a warning icons on all model elements affected. There are two approaches to resolving this reference warnings and the recommended approach depends on who is owner of the referring project. For projects where the modeller is not owner of the referenced model element then the update must be done using the following steps to pull the changes into the project

  • Select the project folder in the Model Explorer view
  • Synchronize the changes by selecting <RMB>>Synchronize with Library

In cases where the modeller is owner of both the referenced and the referring projects then the update can be globally pushed from the referenced project to the referring projects using the following steps

  •  Select the project folder in the Model Explorer view
  • Synchronize the changes by selecting <RMB>>Synchronize references

For either approach, the Undo operation can reverse the changes made during synchronization.

7 Importing masl models into BridgePoint

To translate a MASL into a BridgePoint MASL model involves two steps: a conversion step and a xtUML import step. The conversion step is discussed in section 4.1 and involves the  MASL specific extension, masl2xtuml. The output from this conversion tool is a conventional xtUML model file that can be imported into an existing xtUML project using the standard import facility. BridgePoint xtUML import can be achieved either the command line or interactively through the user interface. Each of these alternative will be described in sections 4.2 and 4.3 respectively.

7.1 MASL Model Conversion

The conversion of MASL domains and MASL projects into xtUML model files is performed by the tool, masl2xtuml and is invoked from the command line of a shell terminal. A MASL model consists of a domain(s) and a project with the associated files for each organized under individual directories. So the translation of a complete MASL model involves the conversion of each directory into an associated xtuml model file that is later import into BridgePoint during the second step. .

The conversion command has the following syntax

masl2xtuml -d <directory path> -o <directory path>

where  <directory path> is any relative or absolute directory path, -d identifies the domain or project directory and -o identifies the directory with the xtUML model file is to be written to. A complete reference is included in the MASL documentation

The MASL domain(s) and MASL project are processed one at a time in any order. All conversion can be performed sequentially or each conversion can be followed by a BridgePoint model import. However, the conversion is incomplete until all MASL domains and MASL  projects have been successfully processed.  

Note that MASL models can be converted in any order, however BridgePoint import does require that all xtUML models representing MASL domains be imported prior to the import of the top-level MASL project model.

8 Exporting xtuml models to masl

The conversion of xtUML models to MASL is a single step operation that can be invoked from either the UI or a terminal shell command line

8.1 Command line

The command xtuml2masl is used to export xtuml models to MASL equivalent models without invoking the BridgePoint UI.

Command line Export makes use of the environment variable WORKSPACE.

It is recommended that all xtUML projects be contained in a single workspace, however this is not enforced by BridgePoint export. If complete xtUML model is contained in  multiple BridgePoint workspaces then the WORKSPACE environment variable must be modified between xtuml2masl invocations.

The export command has the following syntax

xtuml2masl [ -i <directory path > -d <component name> ]  |  [-i <directory path > -p <component name> ]

Where -i indicates the xtuml project, -d indicates the xtUML component will be exported as a MASL domain, and -p indicates the xtUML component will be exported as a MASL project.

Progress and error messages are reported to STD ERR

Output MASL is placed in sub directories under the masl_out directory located in the directory where xtuml2masl is invoked.

Each sub directory will be named according to the xtuml components it corresponds to.

 

8.2 Interactive UI

The steps to export an xtUML model to MASL are invoked from the Model Explorer view

Expand the project of the domain or project to be converted

Select the package or component located within that project. There should be only one.

The conversion is activated by selecting <RMB>>Export MASL domain if a domain project or <RMB>>Export MASL project if the xtUML project is to be a top level MASL project.

During the export process, progress is reported in the Console view
