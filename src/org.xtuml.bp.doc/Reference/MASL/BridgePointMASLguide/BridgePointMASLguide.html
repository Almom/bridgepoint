1 Introduction to BridgePoint MASL

BridgePoint xtUML is designed to support the executable, translatable UML (xtUML) approach to the Shlaer-Mellor methodology for Model-driven Analysis (MDA). However, xtUML is not the only approach to have its roots in this MDA approach. There is another variant known as MASL that offers a text-only approach to Shlaer-Mellor modeling. MASL has the advantage that no special entry tools beyond a basic text editor is needed, and these models are easy to maintain under version control. Yet for complex models, the structure and activity is spread across hundreds of files making it tedious and time consuming work for those unfamiliar with the model to gain a full understanding.  

The xtUML approach on the other hand, combines a structural description captured in graphics with a textual action language for activity descriptions. This combination provides a complete xtUML model that is easier to comprehend while maintaining much of the simplified model entry offered by text. However, specialized graphical editors are required to enter the structural content, and these editors are available from the Open Source Software community..

MASL and xtUML share a common ancestry with the Shlaer-Mellor methodology which provides modelers with the potential to move between the two model representations. BridgePoint xtUML is the reference editing and execution platform for xtUML modelers. By adding conversion tools, extensions to import/export facilities and a MASL-aware editor to BridgePoint xtUML leads to BridgePoint MASL which is a powerful platform for modeling in either language. This gives each group the opportunity to convert existing models into the alternate language, take advantage of the strengths offered while modifying the model and then return it to the original language. Conversion in either direction is possible, however, an idiom must be followed when moving xtUML models into MASL. This page documents that idiom and the extensions to BridgePoint xtUML that make BridgePoint MASL possible.  

2 The MASL - xtUML Modeling Idiom

To ensure that models represented in either xtUML or MASL can consistently move between either Shlaer-Mellor variant, a set of recommendations and restrictions must be followed. Collectively these form the MASL-xtUML Idiom and is the foundation for model representation and error-free model exchange with BridgePoint MASL.

Since MASL and xtUML are based on Shlaer-­Mellor MDA, the two languages are very similar at the core. Consequently, most mappings between constructs in the two languages are straight­forward (e.g., MASL Object maps to xtUML Class) and are therefore not detailed here. However, each language has been extended differently leading to some mappings that are less obvious, and these will be explained in the sections below.

The Idiom addresses three areas of concern: Type mapping and scope, bridges between domains, and model packaging. By following the practices outlined in the MASL-xtUML Idiom. models that start as MASL can be converted to xtUML, modified inside the xtUML Editor and exported back as MASL.

2.1 Types

The MASL type system does not align perfectly with that of xtUML, so the entire MASL type system is adopted into the xtUML Editor. As long as parameters and attributes can be typed with MASL types, the export facility can produce the appropriate MASL output. For BridgePoint MASL  no MASL types are mapped to xtUML types which leaves ­­ the two type systems separate.

2.1.1 Type Reference

Where xtUML typing is established through a link between the model element being typed and a particular instance of a type, MASL treats a type reference as a first­class concept. Typing a model element in MASL is slightly more complex than it is with xtUML, because a type reference can further constrain the type used to type the model element in question. Both attribute and variable declarations can include a range, but parameter declarations cannot. Since variable declarations exist only within activities expressed in MASL and during import are handled separate from OAL, no special handling is required for them

2.1.2 Compound Types

Typing a model element with a collection type or an instance type requires more than the selection of a type. For collection types, the type of collection (sequence, bag, set, array) must be specified along with the type of each element in the collection. Additionally, an array specification must also include the number of elements, and a sequence specification may include an optional expression. For an instance type two pieces of information are required: “instance of” and a class name. To define a collection of instances then requires a declaration of the form: sequence of instance of <className>.

2.2 Model Packaging

User-defined MASL types are defined within a MASL domain, and public types can be accessed by other MASL domains using a qualified name of the form:  otherDomainName::someType.  However, upon conversion to xtUML, each MASL domain is packaged within a component.  Model elements defined within one component have no knowledge or visibility of model elements (including types) defined within another component.  Consequently, including types defined within a MASL domain inside the component representing the MASL domain makes them invisible to other components in the system.  Accordingly, all shared types, including those used on terminator services, must be packaged outside domain components.

Similarly, all interface definitions derived from the public services/functions surfaced by a MASL domain and the terminators declared within it must be packaged outside the component representing the MASL domain.


To address these differences in scoping of types and interface definitions, the Idiom recommends that any MASL domain converted to xtUML contain a single top-level domain package which contains:

  • one component definition representing the MASL domain
  • one package containing all shared types and all interface definitions associated with the MASL domain

A typical xtUML project using this approach is illustrated below. Note that all shared types defined in the MASL domain are contained within the Shared package associated with that MASL domain, and these type definitions are then not included in the component representing the MASL domain. With BridgePoint MASL, the conversion facility produces a single xtUML model file adhering to the packaging and naming scheme illustrated. Modelers are free to repackage and rename their models as they see fit so long as these two conditions are met:

  1. All elements shared among components are defined outside the components that refer to them.
  2. All elements associated with a MASL domain reside within the domain package for that MASL domain.

For xtUML models, packaged in this way, the export facility produces a MASL domain file (.mod) that includes the shared types residing in the package associated with the MASL domain.

2.3 Bridging

Systems are typically constructed by connecting multiple MASL domains or xtUML components using some form of bridging technology. Where MASL employs terminators and public services/functions, xtUML uses interfaces consisting of interface operations/signals and ports.

The basic mappings are simple: a service declared on a terminator corresponds to an interface operation on a port while a public service surfaced by a MASL domain is rendered as a function invoked by a port activity.

MASL bridging provides a good deal of flexibility in the way MASL domains can communicate. For example, a MASL domain may use a subset of the services surfaced by another MASL domain, and it may choose to refer to those services by different names than the MASL domain providing them. Also, a forking bridge allows a MASL domain to view as a single service two or more services provided by any number of MASL domains. Since xtUML provides a slightly less flexible form of bridging based on a single definition for each interface, the idiom illustrated below is used to provide the same level of domain isolation afforded by the MASL approach.

In this diagram, magenta rectangles are references to reusable domain components while the green rectangle is a project­specific bridge specification (project component).

Each component representing a MASL domain sprouts a single provided interface containing an interface operation for each public service/function surfaced by the MASL domain as declared in the MASL domain interface file. This provided interface covers all incoming interface operations for the domain component.

3 Recommendations when Implementing the MASL-xtUML Idiom

Although MASL and xtUML share many similarities, the consistent movement of models between the two variants of Shlaer-Mellor requires models to follow a small set of recommended practices.

These practices fall under the categories of model structure and tool usage.

3.1 MASL Model structure recommendations

MASL models consist of domains and projects

Each MASL domain or project includes several text files describing structure and activity.

It is recommended that a project and its associated domains be organized into directory folders such that each folder contains all files associated with a single domain or top level project.

3.2 xtUML Model structure recommendations

The  organization of xtUML projects inside the eclipse workspace is similar to the organization of filesystem containing MASL domains and MASL projects. Each xtUML project exists as a unique directory directly below the workspace directory.

Each MASL domain is imported into an xtUML project with one MASL domain per project

The xtUML project name should match the MASL domain or MAL project name

The organization of the xtuml model elements within a project package is different for models that contain domain components and those that contain project packages. Common to both is the shared package which contains model elements for all shared types used by this domain component.

4 Conversion Extensions for BridgePoint MASL

5 Appendix a.  the GPS Watch Example

To demonstrate the Import and export extensions of BridgePoint MASL, the GPS_Watch example will be used. This model represents an exercise watch that records location, heart rate, and elapsed time as the wearer performs  a physical activity. The model is composed of four components:  a heart rate monitor, a location component, a tracking component and a user interface.  

5.1 Create a local copy of the MASL model

This model is located in the public GitHub repository at https://github.com/xtuml/models. The local copy we will use as the starting point in this tutorial is produced from a ZIP file image of this repository.

  1. To produce that ZIP file image, click on the Download ZIP button just above the directory listing for the top level of the repository. Shortly after clicking on the button, a download will automatically start.
  2. Open a file browser and create a folder named Tutorial in your home directory
  3. With the same file browser navigate to the folder Downloads where the models-master.zip file should reside.
  4. Extract the contents of models-master.zip to ~/Tutorial/

5.2 Convert the MASL model to xtUML

  1. Copy the contents of ~/Tutorial/model-master/applications/gps/GpSWatchMASL/MASLsrc to ~/Tutorial/MASLsrc
  2. Open a terminal window and navigate to the directory ~/Tutorial/MASLsrc . In this directory you will find the subdirectories GPSWatch, HeartRateMonitor, Location, Tracking and UI.  
  3. To convert each MASL domain and MASL project to an equivalent xtUML model the masl2xtuml tool is used. Each of the subdirectories will be converted individually in any order using this tool and the converted projects placed in ~/Tutorial/xtUMLprj.
  4. The command to convert HeartRateMonitor is
            masl2xtuml -d ./HeartRateMonitor -o ../xtUMLprj
  5. Repeat step d for the other MASL domains and MASL projects in the directory

At the conclusion, the xtUMLprj directory will contain five subdirectories with names matching the five subdirectories under ~/Tutorial/MASLsrc

5.3 Launch BridgePoint MASL

  1. The first step is to launch BridgePoint MASL and select the eclipse workspace to hold the modelling projects.

  1. To launch BridgePoint MASL, execute the launch script <BridgePoint Home>/eclipse/Launcher.sh and during the startup of BridgePoint MASL the Select a workspace pop up window will appear.

  1. In this window, either type the path to the desired workspace directory or click on Browse and navigate to this folder.

  1. Once entered, click on OK.

  1. BridgePoint MASL will then open in the xtUML Modeling perspective. The Model Explorer[a] view should be empty at this point.

5.4 Create xtUML projects

Having produced the individual xtUML projects, they are ready to be assembled into an xtUML model inside BridgePoint MASL[b]. The model structure described in section 3.x will be used here using the following steps

  1. Create an empty xtUML project using File>New>xtUML Project and give it the name HeartRateMonitor
  2. Import the converted xtUML model using File>Import and select xtUML Model under the xtUML folder. Once selected, click next.
  3. Browse to the folder ~/Tutorial/xtUMLprj/HeartRateMonitor and then check mark HeartRateMonitor for the destination
  4. Select the HeartRateMonitor project folder in the Model Explorer view and then select <RMB>>Project Preferences. This will popup the Preferences window and after selecting Inter-project References should confirm that the preference Allow inter-project model references is checked.
  5. Repeat steps a. Through d. for Location, Tracking and UI
  6. The final step is to import the top level project GPSWatch, Again create an empty xtUML project using File>New>xtUML, give it the name GPSWatch, and confirm that the preference Inter-project model References is checkmarked.
  7. Import the converted MASL project found in ~/Tutorial/xtumlprj/GPSWatch into the GPSWatch xtUML project.

5.5 Navigating through xtUML model

With the model now populated inside the workspace, the various views and editors can be used to visualize, modify and export the model. Navigating around the model can be performed from inside either the Model Explorer or Navigator views. Each view employs a tree view of the model and individual model elements can be located using the following steps

  1. e[c]xpand the tree from the project folder down through the model packages by clicking the triangle next to the right of the package or model element symbol.
  2. Repeat until the desired element is revealed in the tree.
  3. To open the model element in a BridgePoint editor, hover over element and then <dbl click>.

For MASL activity blocks, the Navigator view can be used launch the BridgePoint MASL editor using a similar sequence of steps

  1. b[d]y expanding the tree from the project folder down through the model packages and model elements
  2. Repeat until the desired .masl file is revealed in the tree.
  3. To open the file in a BridgePoint MASL editor, hover over filename and then <dbl click>.

5.7 Exporting the xtUML model to MASL domains and MASL projects

The conversion of xtUML projects into MASL domains and MASL projects is conveniently performed from the Model Explorer view by first selecting the top package within the xtUML project .

  1. Begin with the xtUML project HeartRateMonitor which is a domain within the GPSWatch model.
  2. Once the package is selected, convert the project into a MASL domain by <RMB>>Export MASL domains
  3. Check the Console view for Errors or INFO messages. A masl directory will now appear in the directory tree of the project.
  4. Repeat steps b and c for xtUML projects UI, Location, and Tracking.
  5. The final conversion is for the GPSWatch project which will be converted into a MASL project. Select the GPSWatch package and <RMB>>Export MASL project
